Claude.ai vs Cursor

仕組みの違いを原理から理解し、どちらをいつ使うか自分で判断できる状態になる

ざっくり一言: Claude.aiは対話型AI、CursorはAIが動くコードエディタ

設定のイメージ | 全体像 | 3つの構造 | Claude.ai | Cursor | 比較 | ガバナンス | 使い分け | まとめ

この図解が答える問い

Claude.aiとCursorは
何が同じで、何が違うのか。

どちらもAIを使うツールだが、アーキテクチャ・コンテキストの取り込み方・実行できる操作が根本的に異なる。仕組みを理解し、「このタスクはどちらで解くべきか」を自分で判断できる状態になる。

設定変更のイメージ

どこを設定すると何が変わるか

「使い分ける」と言っても、実際にどの設定を触ると何が変わるのかが分からないと判断できない。ツール別の主要設定ポイントと変更後の状態を整理する。

Claude.ai の設定ポイント
  • 1

    Projects にドキュメントをアップロードする

    変更前: 会話ごとにファイルを貼り付ける必要あり。変更後: プロジェクト内でドキュメントが永続化され、RAG で自動参照される。社内規程・ベンダー仕様書・過去の障害記録を入れておくと横断的な質問ができる。 ↗ Projects 公式ヘルプ

  • 2

    Project instructions（指示書）を設定する

    変更前: 毎回「コーポレートエンジニア視点で」と前置きが必要。変更後: 指示書に書いた内容がデフォルトの振る舞いになる。例: 「回答は設定変更の根拠と変更後の状態を必ず含めること」。 ↗ Projects 公式ヘルプ

  • 3

    モデルを切り替える（Sonnet vs Opus）

    変更前 → 後: Sonnet は速度重視・日常の質問向け。Opus は複雑な推論・設計判断向け。設定変更の根拠となる技術的な深い説明が必要な場面では Opus を選ぶ。 ↗ Claude モデル一覧（Anthropic 公式）

Cursor の設定ポイント
  • 1

    バックエンドモデルを選択する

    変更前 → 後: 設定画面でモデルを切り替えると、以降の Agent / Chat がそのモデルで動く。Claude Sonnet（コスパ） / Opus（深い推論） / GPT-4o など目的で切り替え可。 ↗ Cursor 公式（モデル選択の言及あり）

  • 2

    Agent モード vs Chat モードを選ぶ

    Chat: 提案だけ（ファイルは変更されない）。Agent: ファイルを実際に作成・編集・削除する。ターミナル実行も可。大きな変更前は Plan モード（Shift+Tab）でエージェントに計画を立てさせてから承認する。 ↗ Agent モード公式 docs ↗ Plan Mode 公式ブログ

  • 3

    Privacy Mode をオン/オフにする

    Privacy Mode OFF（デフォルト）: コードが AI 改善のために使われる可能性あり。Privacy Mode ON: モデルプロバイダーへのデータ保持ゼロ。コードが学習に使われない。社内コードを扱う場合は原則 ON にする。 ↗ Cursor データ利用・プライバシー（公式）

まず理解する

3つの構造

この順に理解すると「なぜ使い分けが必要か」が腹落ちする。

01
骨格

「モデル」と「製品」は別物

Claude は Anthropic が開発した AI モデル。Claude.ai はそのモデルを包んだ会話 UI（製品）。Cursor は別会社（Anysphere）が作ったコードエディタで、バックエンドのモデルとして Claude を「選択できる」関係にある。Claude.ai で Claude を使うことと、Cursor で Claude を選ぶことは全く別の体験になる。

↗ Anthropic Claude 公式 ↗ Cursor 公式
02
補足

コンテキストの取り込み方が根本的に違う

Claude.ai はアップロードしたドキュメントや会話履歴を参照する。Cursor はローカルのコードベース全体をインデックスし、セマンティック検索で関連箇所を自動取得する。「何を知っているか」の源泉が異なるため、同じ Claude モデルでも回答品質は異なる。

↗ Claude Projects RAG（公式ヘルプ） ↗ Cursor Product ページ
03
応用

エージェントが実行できる操作の範囲が違う

Claude.ai（Cowork）はローカルファイルやクラウドアプリと連携してタスクを進める。Cursor のエージェントはファイル編集・ターミナル実行・git 操作まで行う。Cursor のエージェントはコードの変更を実際に適用するが、Claude.ai は基本的に提案止まり（Cowork 除く）。

↗ Claude Cowork（Anthropic 公式） ↗ Cursor Agent（Product ページ）

Claude.ai の仕組み

AIとの対話ツール

Anthropic が提供する Web・モバイルの会話インターフェース。「問いを投げて答えを得る」対話が基本。Project 機能でナレッジを蓄積し、RAG（Retrieval Augmented Generation＝検索して関連情報を取得）で大量ドキュメントを参照可能。

↗ Claude Projects 公式ヘルプ
  • 1

    会話 → Claude モデルが応答

    ブラウザやアプリからメッセージを送信。Anthropic サーバ上の Claude モデル（Opus / Sonnet / Haiku）が応答を生成。コンテキストウィンドウは有料プランで 200K トークン。 ↗ コンテキスト公式ヘルプ

  • 2

    Projects + RAG でナレッジを拡張

    プロジェクトにドキュメントをアップロードしておくと、会話内容に応じて関連箇所をセマンティック検索して取得（RAG）。コンテキスト上限に近づいても全量を詰め込まず検索で補う。プロジェクト指示（instructions）でClaude の振る舞いを固定できる。 ↗ RAG 公式ヘルプ

  • 3

    Cowork: ローカルファイルへの直接アクセス（別プロダクト）

    Anthropic の Cowork デスクトップアプリは、ローカルファイルへの直接アクセス・長時間タスクの自律実行・並列ワークストリームの調整が可能。隔離された仮想マシン上で動作し、セキュリティを担保。現在はリサーチプレビュー段階（有料プランのみ）。通常の claude.ai チャットとは別プロダクト。 ↗ Claude Cowork 公式ページ ↗ Cowork はじめ方（公式ヘルプ）

得意なこと

  • テキスト生成・説明・要約・翻訳などの言語タスク
  • 大量ドキュメントの読み込みと Q&A（Projects + RAG）
  • 調査・分析・戦略の検討
  • コードを「書いてもらう」（実行はしない）
  • SaaSの設定方法を理解するための対話

できないこと / 注意点

  • ローカルのファイルシステムへのアクセス（Cowork 除く）
  • コードの実行・テスト・デプロイ（Cowork 除く）
  • コードベース全体をコンテキストとして扱う自動索引化
  • 出力は常に正確とは限らず検証が必要 ↗ 利用規約

Cursor の仕組み

AIを内蔵したコードエディタ

Anysphere が開発した VS Code ベースの IDE。コードベース全体をインデックスし、AI がファイル編集・ターミナル操作・デバッグまで自律的に実行できる。バックエンドの AI モデルは Claude / GPT-4 / Gemini など複数から選択可能。

↗ Cursor Product ページ（公式）
  • 1

    コードベース全体をインデックス

    プロジェクトのコードベースをカスタム埋め込みモデルでインデックス化。セマンティック検索で関連コードを自動取得してからAIに渡す。「このファイルどこだっけ」を毎回 @指定しなくても文脈を理解できる。

  • 2

    エージェントがファイルを直接編集

    Agent モードでは提案だけでなく、ファイルの作成・変更・削除を実際に適用する。Git チェックポイントで変更前の状態に戻ることが可能。ターミナルでビルド・テストも実行できる（デフォルトはサンドボックス）。

  • 3

    Plan モードで計画→承認→実行のサイクル

    Shift+Tab で Plan モードに入るとエージェントがコードベースをリサーチし、変更計画（Markdown）を先に提示してから実行。複数ファイルにまたがる変更や要件が不明確な時に安全に使える。计画を確認・編集してから「Build」で実行する。 ↗ Agent モード公式 docs ↗ Plan Mode 公式ブログ

Cursor の特徴的な強み

✓

コードベース全体を「知っている」状態でAIが動く

Claude.ai に毎回ファイルを貼り付けなくても、既存コードの文脈を踏まえた実装・修正ができる。

✓

モデルを選択できる

Claude / GPT-4o / Gemini / Grok など複数社のモデルを切り替え可能。タスクや料金に応じて使い分けられる。 ↗ Cursor 公式

✓

変更をGitでロールバックできる

エージェントの変更はチェックポイントとして記録。実行前の状態に戻せるため、AIの変更を安全に試せる。

コーポレートエンジニア必須の確認事項

データ・プライバシー・ガバナンス

社内コードや機密ドキュメントを扱う場合、「そのデータがどこに送られ、学習に使われるか」の確認は上長への説明責任として必須。両ツールの公式ポリシーから言えることを整理する。

Claude.ai のデータ扱い

  • Projects のドキュメントはサーバに保存される

    アップロードしたドキュメントは Anthropic のサーバに保存される。機密性の高い情報の取り扱いはプランと契約内容を確認すること。

    ↗ Anthropic 利用規約

  • Team / Enterprise プランはデータ保護が強化される

    エンタープライズ契約では学習への使用制限・データ保持ポリシーが変わる場合がある。※ 詳細は契約時に要確認

    ↗ Anthropic 公式（エンタープライズ情報あり）
Cursor のデータ扱い

  • Privacy Mode ON = モデルプロバイダーへのデータ保持ゼロ

    Privacy Mode を有効にするとコードが学習に使われない。コードベースのインデックス時、埋め込み計算用のコードは各リクエスト後に削除される（埋め込みとメタデータは保持）。

    ↗ Cursor データ利用・プライバシー（公式）

  • コードは Cursor 経由でモデルプロバイダーに送信される

    Cursor で Claude を使う場合、コードは Anthropic ではなく Cursor（Anysphere）サーバを経由してモデルに送られる。Anthropic の AUP ではなく Cursor の ToS が適用されることに注意。

    ↗ Cursor プライバシーポリシー（公式）

社内導入前に確認すべき事項（別途確認が必要）

  • → 機密区分の社内コード・顧客データを入力してよいか（情報セキュリティポリシーとの照合）
  • → エンタープライズ/チームプランの契約有無と DPA（データ処理契約）の状況
  • → 送信先サーバのリージョン（米国サーバか否か）と社内の海外データ移転ポリシー

※ この図解が扱う範囲外。各社の法務・情報セキュリティ部門への確認を推奨。

コーポレートエンジニア視点

どちらをいつ使うか

「どちらが優れているか」ではなく「タスクの性質でどちらを使うか」を判断できることがゴール。

Claude.ai を使う場面

  • 技術の仕組みを理解したいとき

    RADIUSのフェイルオーバーの原理・SaaSの設定値の意味など、「なぜそうなるか」を対話で理解するのに向いている

  • 大量ドキュメントを読み込ませて質問したいとき

    仕様書・ベンダーマニュアル・障害記録をプロジェクトにアップロードし、横断的に質問できる

  • 文章を作成・要約・翻訳するとき

    提案書・障害報告・社内メール・設定変更の影響説明など

  • コードを相談・設計レビューするとき

    「このスクリプトのアプローチが正しいか」という判断を対話で確認する用途

Cursor を使う場面

  • 既存コードベースを修正・機能追加するとき

    コードベース全体を把握した状態でAIが変更を適用するため、「どのファイルを変えるか」を毎回指示しなくてよい

  • ビルド・テスト・git を繰り返しながら開発するとき

    ターミナル実行 → エラー確認 → 修正 → 再実行のサイクルをエージェントが一括して回せる

  • スクリプトや自動化ツールをゼロから作るとき

    Claude.ai で設計・仕様を固めてから Cursor で実装するという2段階が効率的

  • 大きなリファクタリングをするとき

    変更前のチェックポイントを残しながらエージェントに複数ファイルを横断した変更を任せられる

2つを組み合わせる実務パターン

Step 1

Claude.ai で「理解」 → 技術の仕組みを対話で把握し、設定値の根拠・変更後の状態を確認する

Step 2

Cursor で「実装」 → 理解した設定・スクリプトを実際のコードベースに適用・テスト

Step 3

Claude.ai で「説明」 → 実施した変更の根拠・影響を上長向けに整理・言語化する

まとめ

この図解でわかること

01

モデルと製品は別物

Claude は Anthropic のモデル。Claude.ai はそのUIで、Cursor はモデルを選べるエディタ。「Cursor で Claude を使う」≠「Claude.ai を使う」。コンテキストと操作範囲が根本的に異なる。

02

コンテキストの違いが出力品質を決める

Claude.ai はアップロードしたドキュメントのみ。Cursor はコードベース全体をインデックスして関連箇所を自動取得。「何を知っているか」の違いが同一モデルでも回答品質に直結する。

03

実行できる操作の範囲が違う

Claude.ai は基本的に提案止まり。Cursor のエージェントはファイル編集・ターミナル実行・git まで自律実行できる。「話して理解する」と「コードとして適用する」で使い分けるのが合理的。

この理解で次にできること

→

技術調査と実装の場面でどちらを起動するか迷わずに判断できる

→

「Claude.ai でわかったこと」を根拠に Cursor でコードを変更し、変更の正しさを上長に説明できる

→

「Cursor でどのモデルを使うべきか」「Claude.ai の Projects に何をアップロードすべきか」を設計できる
