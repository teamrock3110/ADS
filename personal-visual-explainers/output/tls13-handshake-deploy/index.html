TLS 1.3 ハンドシェイク

ざっくり一言

TLS 1.3は1往復でセッション鍵を確立し、過去通信の復号を不可能にした暗号プロトコル

ハンドシェイクの各メッセージが何をしているかを順序通りに理解し、「なぜTLS 1.3が安全なのか」を自分の言葉で説明できる状態になる。

この図解が答える問い

ブラウザとサーバはHTTPS通信で何を何回やりとりして、いつ暗号化が始まるのか。

全体像

TLS 1.3 ハンドシェイク（1-RTT）

ブラウザ（クライアント）とサーバが1往復（1-RTT）でセッション鍵を確立する。ServerHello以降のメッセージはすでに暗号化されている。

【TLS 1.3: 1-RTT ハンドシェイク】

[クライアント]                              [サーバ]
      |                                        |
      |  ClientHello                           |
      |  (key_share: ECDHEの公開鍵, 対応暗号)  |
      |--------------------------------------->|
      |                                        |
      |  ServerHello                           |
      |  (key_share: サーバの公開鍵を返す)     |
      |<---------------------------------------|
      |  {EncryptedExtensions}  ← ここから暗号化
      |<---------------------------------------|
      |  {Certificate}                         |
      |<---------------------------------------|
      |  {CertificateVerify}                   |
      |<---------------------------------------|
      |  {Finished}                            |
      |<---------------------------------------|
      |                                        |
      |  {Finished}                            |
      |--------------------------------------->|
      |                                        |
      |  === アプリケーションデータ（暗号化） ===

  ※  { } = Handshake Keysで暗号化 / [ ] = Application Keysで暗号化
  ※  key_shareの交換完了後に2段階で鍵を導出する（下図）

【鍵の2段階導出（RFC 8446 §7.1 Key Schedule）】
  key_share交換
      ↓
  ┌── Handshake Keys ──────────────────────────────────────┐
  │ EncryptedExtensions・Certificate・CertificateVerify     │
  │ Finished の暗号化に使用（ハンドシェイク中のみ有効）       │
  └────────────────────────────────────────────────────────┘
      ↓ 双方のFinished完了後に再導出
  ┌── Application Keys ────────────────────────────────────┐
  │ HTTP等のアプリケーションデータの暗号化に使用              │
  │ （Handshake Keysと異なる鍵なので独立した保護）           │
  └────────────────────────────────────────────────────────┘

【TLS 1.2との往復回数比較】
  TLS 1.2: 2-RTT（鍵交換 → 認証 → 別途Finishedの往復）
  TLS 1.3: 1-RTT（ClientHello/ServerHelloで鍵交換を同時に完了）
  TLS 1.3 0-RTT: 0往復（再接続時にセッションチケットで即データ送信）
           └─ ただし0-RTTはリプレイ攻撃のリスクあり（RFC 8446 §8）
key_share = ECDHE鍵交換に使う公開鍵 RTT = Round-Trip Time（1往復） ECDHE = 楕円曲線Diffie-Hellman鍵交換（使い捨て）

まず理解する

3つの構造

1
鍵交換と認証は別の操作 骨格

TLSのハンドシェイクは「①誰と話しているか確認（認証）」と「②盗聴されない鍵を作る（鍵交換）」の2つを同時に行う。TLS 1.3ではClientHelloとServerHelloの1往復でECDHEによる鍵交換を完了させ、認証（証明書検証）はその後に暗号化された通信路の中で行う。

↗ RFC 8446 §2 Protocol Overview
2
ECDHE が前方秘匿性を保証する 補足

TLS 1.2まで使えたRSA静的鍵交換では、サーバの秘密鍵が漏洩すると過去の全通信が復号できた。TLS 1.3はECDHE（楕円曲線Diffie-Hellman鍵交換・使い捨て）のみを許可し、セッションごとに使い捨ての鍵ペアを生成するため、秘密鍵が漏洩しても過去通信は復号できない（前方秘匿性）。

↗ RFC 8446 §1.2 Major Differences from TLS 1.2
3
0-RTT は利便性とリスクのトレードオフ 応用

再接続時にセッションチケット（PSK＝事前共有鍵）を使うと、ハンドシェイクなしで即データを送信できる（0-RTT）。ただし0-RTTデータはリプレイ攻撃に脆弱であり、同じデータが複数回実行される可能性がある。GETリクエストのような冪等操作のみに限定することが推奨される。 ↗ RFC 8446 §8

鍵交換の原理

なぜ公開情報だけで秘密を共有できるか

ECDHE（楕円曲線Diffie-Hellman鍵交換・使い捨て）が「なぜ盗聴されても安全なのか」を直感的に理解する。

絵の具アナロジー（数式なし）

①

AとBが「共通の黄色（公開パラメータ）」を持っている。これはネットワーク上に公開されても構わない。

②

Aは自分の「秘密の赤」をこっそり持ち、黄色に混ぜて「オレンジ」を作ってBに送る。Bは自分の「秘密の青」を黄色に混ぜて「緑」を作ってAに送る。

③

AはBの「緑」に自分の「秘密の赤」を混ぜると → 黄+青+赤 になる。Bは「オレンジ」に自分の「秘密の青」を混ぜると → 黄+赤+青 になる。混ぜる順序が違っても結果が同じ色になるのが鍵交換の本質。

④

盗聴者が見えるのは「オレンジ」と「緑」だけ。そこから元の「秘密の赤」や「秘密の青」を取り出すことは、楕円曲線の数学的特性により計算量的に不可能とされている。

TLSでの対応

共通の黄色 = 合意した楕円曲線パラメータ（公開）
秘密の赤・青 = 各自が生成した一時秘密鍵（非公開）
オレンジ・緑 = key_shareで交換するECDHE公開鍵
黄+赤+青の色 = 共有シークレット（ここからHandshake Keysを導出）

「使い捨て（Ephemeral）」がつく理由: TLS 1.3ではこの一時鍵ペアを接続ごとに毎回生成して破棄する。そのため長期秘密鍵（サーバ証明書の秘密鍵）が漏洩しても過去の接続は復号できない（前方秘匿性）。 ↗ RFC 8446 §4.2.8 key_share拡張

ハンドシェイク詳細

各メッセージの役割

ServerHelloまでがクリアテキスト（平文）で、それ以降はHandshake Keysで暗号化される。鍵の導出はHKDF（HMAC-based Key Derivation Function）で行う。

↗ RFC 8446 §7 Cryptographic Computations
1

ClientHello — クライアント → サーバ

対応する暗号スイート・TLSバージョン・key_share（クライアントのECDHE公開鍵）を送信。ここで対応暗号の候補を全て列挙する。

2

ServerHello — サーバ → クライアント

選択した暗号スイートとサーバのkey_share（ECDHEの公開鍵）を返す。この交換でお互いが共有シークレットを計算でき、以降は暗号化通信に切り替わる。

3

{EncryptedExtensions} — サーバ → クライアント（暗号化）

認証に関係しない追加の拡張情報を暗号化した状態で送信。TLS 1.2ではこれらが平文だったため、通信のメタデータが盗聴可能だった。

4

{Certificate} — サーバ → クライアント（暗号化）

サーバのX.509証明書を送信。クライアントはこれを使ってサーバが信頼できる認証局（CA）の証明書チェーンを持つか確認する。

5

{CertificateVerify} — サーバ → クライアント（暗号化）

証明書に記載された公開鍵に対応する秘密鍵を本当に持っているかをデジタル署名で証明する。ハンドシェイク全体のトランスクリプトに署名するため改ざん検知も兼ねる。 ↗ RFC 8446 §4.4.3

6

{Finished} — 双方向（暗号化）

ハンドシェイク全体のMAC（メッセージ認証コード）を送り合い、完了を確認する。両者のFinished検証が成功した時点でアプリケーションデータの送受信を開始できる。 ↗ RFC 8446 §4.4.4

TLS 1.2との違い

廃止されたものと追加されたもの

TLS 1.3はレガシーな暗号を廃止して攻撃面を削減した。 ↗ RFC 8446 §1.2 Major Differences from TLS 1.2

TLS 1.3で廃止

RSA静的鍵交換

サーバ秘密鍵の漏洩で過去通信を全復号できるため廃止。ECDHEのみ許可。

静的DH鍵交換

使い捨てでないDHも廃止。前方秘匿性のない鍵交換方式を全て排除。

非AEAD暗号（CBC等）・RC4・3DES

TLS 1.3ではAEAD暗号のみ許可（§1.2）。CBCモードはTLS 1.3の暗号スイートとして仕様上定義されておらず、BEAST・Lucky 13等の既知攻撃への対策も兼ねる。

MD5・SHA-1 ベースの署名

衝突攻撃が実証されたハッシュ関数を廃止。SHA-256以上のみ許可。

TLS 1.3で追加・変更

1-RTT ハンドシェイク

ClientHello/ServerHelloで鍵交換を完了。TLS 1.2の2-RTTから削減。

EncryptedExtensions

TLS 1.2では平文だった追加情報（SNIのレスポンス等）を暗号化。

0-RTT セッション再開

NewSessionTicket（PSK）を使った0往復の再接続。リプレイ攻撃のリスクあり。

AEAD 暗号のみ許可

AES-GCM・ChaCha20-Poly1305等の認証付き暗号のみ。

コーポレートエンジニア視点

設定値と原理の対応

この図解の原理が、実際の設定ファイルのどの行に対応するかを示す。「なぜこう設定するか」の根拠として使える。

nginx

ssl_protocols TLSv1.3;
ssl_ciphers   TLS_AES_256_GCM_SHA384:TLS_CHACHA20_POLY1305_SHA256;
ssl_ecdh_curve X25519:secp256r1;
ssl_protocols TLSv1.3 → TLS 1.3のみ許可。TLS 1.2以前のClientHelloには応答しない（静的RSA鍵交換・CBC暗号等を根絶）
ssl_ciphers → TLS 1.3の暗号スイートはすべてAEAD。CBC系は仕様上TLS 1.3に存在しないためここでの排除は不要だが、明示しておくと意図が分かる
ssl_ecdh_curve → ECDHE（key_share）で使う楕円曲線の優先順位。X25519はRFC 8446が推奨する曲線 ↗ RFC 8446 §4.2.7

Apache (mod_ssl)

SSLProtocol -all +TLSv1.3
SSLOpenSSLConfCmd Curves X25519:P-256
-all +TLSv1.3 → TLS 1.0〜1.2を無効化しTLS 1.3のみ有効にする。nginxと同様、旧鍵交換方式を排除する意味
Curves X25519:P-256 → nginx の ssl_ecdh_curve と同義。key_shareで使う曲線の優先順位

AWS CloudFront（セキュリティポリシー）

TLSv1.2_2021 / TLSv1.2_2022 → TLS 1.2と1.3の両方をサポート（古いデバイス対応）。TLS 1.3対応クライアントは自動でTLS 1.3を使用する ↗ CloudFront サポートプロトコル
SecurityPolicy の選択基準 → 対応デバイスの最古のOSがTLS 1.3非対応（Android 4.x等）ならTLSv1.2_2021を選択。社内管理デバイスのみなら TLS 1.3のみに絞れる（攻撃面の削減）

設定変更の判断フロー

1. 接続元デバイスの最古OSを確認 → TLS 1.3対応か確認（iOS 13以降・Android 10以降・Windows 10 1903以降で概ね対応）

2. 対応済みなら ssl_protocols TLSv1.3 に絞る → 静的RSA・CBC等の攻撃面を根絶できる

3. 非対応デバイスが残るなら TLS 1.2+1.3 の共存 → ただしTLS 1.2側は前方秘匿性が保証されない設定が残るリスクがある

まとめ

この図解でわかること

01

1-RTTで何が起きているか

ClientHello/ServerHelloでECDHEの公開鍵を交換し、その場でセッション鍵を導出。以降のメッセージはすべて暗号化された中で認証が完了する。

02

なぜ前方秘匿性があるか

ECDHEはセッションごとに使い捨ての鍵ペアを生成する。サーバの長期秘密鍵が漏洩しても、過去のセッション鍵は導出できない（RFC 8446 §1.2）。

03

0-RTTの使いどころと限界

再接続の高速化に有効だが、リプレイ攻撃のリスクがあるため冪等なGETリクエスト以外への適用は慎重に（RFC 8446 §8）。

この理解で次にできること

→

「なぜHTTPSは安全なのか」をハンドシェイクの仕組み（鍵交換と認証の分離・ECDHEによる前方秘匿性）から説明できる

→

TLS 1.3に移行する理由（RSA静的鍵交換・CBC暗号の廃止・1-RTT化）を根拠として挙げて、移行方針の提案ができる

→

0-RTTを採用するシステムについて、リプレイ攻撃のリスクと適用すべきリクエスト種別の制限を把握して設計レビューに参加できる
