全体像

認証ハンドシェイクの比較

WPA2（左）はhandshake中のパケットをキャプチャするだけでオフライン辞書攻撃が可能。WPA3（右）は鍵交換の設計を変え、それを原理的に不可能にした。

【WPA2-Personal: PSK + 4-way handshake】

[クライアント]                   [AP]
      |                           |
      |  Msg1: ANonce             |
      |<--------------------------|
      |  Msg2: SNonce + MIC(PSK)  |
      |-------------------------->|   ← ここをキャプチャして辞書攻撃可能
      |  Msg3: GTK + MIC          |
      |<--------------------------|
      |  Msg4: ACK                |
      |-------------------------->|
      |  === データ通信 (CCMP) ===|

  ⚠  PSKを知っていればhandshakeキャプチャで
     オフライン辞書攻撃が可能（IEEE 802.11i-2004 による設計上の特性）

【WPA3-Personal: SAE (Dragonfly Key Exchange)】

[クライアント]                   [AP]
      |                           |
      |  SAE Commit               |
      |<=========================>|  ← 双方向・同時に計算
      |  SAE Confirm              |   パスワード自体は送受信しない
      |<=========================>|
      |  4-way handshake (PTK生成)|
      |<=========================>|
      |  === データ通信 (CCMP) ===|

  ✓  毎回異なるPMKを生成 → 前方秘匿性あり
  ✓  オフライン辞書攻撃に耐性（RFC 7664 §1.2.2）

まず理解する

3つの構造

この順に理解すると全体像がつながる。

01
骨格

PSK vs SAE
認証方式の根本的な違い

WPA2のPSK方式はhandshakeをキャプチャするだけでオフライン辞書攻撃が可能。WPA3のSAE（Simultaneous Authentication of Equals＝対等な同時認証）は攻撃者が毎回オンラインで試す必要があり、高速総当たりが不可能になる。パスワードが弱くても攻撃コストが上がる。

↗ Wi-Fi Alliance Security ↗ RFC 7664 Dragonfly §1.2.2
02
補足

PMF必須化
管理フレームの保護

Wi-Fiの「切断命令」などの管理フレームはWPA2では任意保護（802.11w）。WPA3ではPMF（Protected Management Frames＝管理フレーム保護）が必須となり、偽の切断命令による強制切断攻撃への耐性が向上する。

↗ IEEE 802.11w-2009 ↗ Wi-Fi Alliance（PMF必須の記載）
03
応用

OWE
オープンネットワークの暗号化

フリーWi-Fiのような認証なしネットワークでも、WPA3のOWE（Opportunistic Wireless Encryption＝機会的無線暗号化）を使うと通信が暗号化される。ただしOWEはAPの認証を提供しないため、偽AP（MITM攻撃）への対策は別途必要。

↗ RFC 8110 §7（OWEのセキュリティ上の制限）

WPA2の認証

PSK + 4-way handshake

WPA2-PersonalはPSK（Wi-Fiパスワード）という事前に共有した鍵を双方が持ち、4-way handshakeで「同じPSKを知っているか」をMIC（Message Integrity Code＝改ざん検知コード）で検証しながらセッション鍵を生成する。 この方式の根本問題は、EAPOL（EAPをLAN上で運ぶフレーム形式）をキャプチャするだけで、接続なしにオフラインでPSK候補を検証できる点にある。GPUを使えば高速に総当たりできる。

↗ IEEE 802.11i-2004（4-way handshake・CCMP・802.1X認証の定義）
  • 1

    AP → クライアント: ANonce

    APが生成した使い捨て乱数（ANonce）を送信

  • 2

    クライアント → AP: SNonce + MIC ← 攻撃者はここをキャプチャする

    クライアントが生成した乱数（SNonce）とMICを送信。このフレームをオフラインで解析することで辞書攻撃が可能。 PSK候補ごとに「このMICになるか」をオフラインで計算できる。

  • 3

    AP → クライアント: GTK + MIC

    GTK（Group Temporal Key＝ブロードキャスト用グループ鍵）とMICを送信。PTKが確定する。

  • 4

    クライアント → AP: ACK

    以降 AES-CCMPで暗号化通信開始

脆弱ポイント

  • オフライン辞書攻撃

    EAPOLフレームをキャプチャするだけで、あとはオフラインGPUで高速に総当たり可能

  • KRACK（鍵の再インストール攻撃）

    4-way handshake中にPTKを再インストールさせnonce再利用で通信を復号。ファームウェア更新で対処可能（Wi-Fi Alliance公式認定）。 ↗ CVE-2017-13077 ↗ Wi-Fi Alliance対応声明

  • 前方秘匿性なし

    PSKが一度漏洩すると、過去にキャプチャしていたすべての通信を復号できる

  • PMFが任意（IEEE 802.11w）

    管理フレームが保護されないため、偽のDeauthフレームによる強制切断攻撃が可能

WPA2が現在も使われる理由

  • 2018年以前の多くのデバイスはWPA3非対応

  • WPA3のTransition Mode（WPA2/3共存）で段階的な移行が可能 ↗ Wi-Fi Alliance

  • 企業（802.1X）環境では攻撃耐性の多くを認証サーバ・証明書で別途担保できる

  • Wi-Fi Alliance はWPA3のみの新規機器購入を推奨 ↗ Wi-Fi Alliance FAQ

WPA3の認証

SAE — Dragonfly Key Exchange

SAE（対等な同時認証）はDragonfly（楕円曲線ベースのパスワード認証鍵交換プロトコル）とも呼ばれる。「パスワードから鍵を計算する」プロセスを双方が同時に行い、その一致を確認する設計。

オフライン攻撃に強い理由（RFC 7664 §1.2.2）: SAEのCommit/Confirmフレームにはパスワードを直接検証するための情報が含まれない。攻撃者が1つの候補パスワードを試すには毎回実際にAPとのプロトコル交換を完了させる必要があり、高速総当たりが原理的に不可能になる。
ただし（RFC 7664 §4）: 能動的な繰り返し推測は可能。パスワードが十分に強くない場合、時間をかけて攻撃される可能性は残る。実装側での失敗回数制限・ブロック等の対策も重要。

↗ RFC 7664 Dragonfly Key Exchange（§1.2.2 辞書攻撃耐性, §4 セキュリティ上の注意）
  • 1

    Commit Exchange（双方向・同時）

    双方がパスワードから数学的に変換したスカラー値・楕円曲線要素を交換。パスワード自体は送受信されない。

  • 2

    Confirm Exchange（双方向）

    双方が確認値を交換し「同じパスワードを持っているか」を検証。PMKが確定。セッションごとに異なるPMKを生成 → 前方秘匿性を実現。

  • 3

    4-way handshake（PTK生成）

    SAE完了後、WPA2と同じ4-way handshakeでPTKを生成し暗号通信開始。KRACKへの対策としてファームウェア更新は引き続き重要。

WPA3で解決された点

✓

オフライン辞書攻撃への耐性

パケットキャプチャだけでは攻撃不可。毎回オンラインで試す必要があり、高速総当たりが不可能になる。 ↗ RFC 7664 §1.2.2

✓

前方秘匿性

セッションごとに異なるPMKを生成。PSKが将来漏洩しても過去通信は復号不可。

✓

PMF必須化

すべてのWPA3ネットワークでPMFが必須。PMFが保護するのは管理フレーム（Deauthentication・Disassociation等）であり、偽の切断命令による強制切断攻撃への耐性が向上する。なおユーザーデータの機密性・完全性はCCMPが担保する（PMFとは別の役割）。 ↗ Wi-Fi Alliance ↗ IEEE 802.11w

コーポレートエンジニア視点

移行ポイント

社内Wi-FiはWPA2/3-Enterprise（802.1X + EAP + RADIUS）で運用するのが標準。WPA3-Enterprise（通常モード）への移行では、EAPの認証フロー自体は変わらないためRADIUSサーバ側の設定変更は一般的に不要。変更が必要なのはAPとクライアントデバイス側のみ。

↗ Wi-Fi Alliance WPA3プレスリリース ※ 通常モード限定の推奨（環境依存あり・IEEE 802.11本文との突合が必要）

移行時の注意点

  • デバイス対応確認が必須

    2018年以前の多くのデバイスはWPA3非対応。移行期はTransition Mode（WPA2/3共存移行モード）を使う。ただしTransition Mode中はWPA2クライアントとの接続も許可されるため、WPA2の攻撃面が残ることを把握した上で期間設計をする必要がある。 ↗ Wi-Fi Alliance

  • 192-bitモードは別途確認が必要

    WPA3-Enterprise 192-bitモード（CNSA＝米国政府推奨の高強度暗号スイート）に移行する場合、EAPメソッドや証明書の暗号アルゴリズム変更が必要な場合がある。 ↗ Wi-Fi Alliance

用途別アドバイス

  • ゲストSSIDにOWEを活用できる

    オープンSSIDをOWEに移行することで無線区間の通信を暗号化できる。ただしOWEはAPの認証を提供しないため、偽APへの対策は別途必要。また旧OSSIDとの並走期間を設けることが推奨される。 ↗ RFC 8110 §6, §7

  • 社内Wi-FiはEnterpriseが適切

    WPA3-PersonalのSAEのメリットは主に家庭・IoT・共用PSKで運用していた環境向け。社内では802.1Xで運用する方がユーザーごとの認証管理ができる。

まとめ

この図解でわかること

01

WPA2の原理的欠陥

PSKはEAPOLフレームをキャプチャするだけでオフライン辞書攻撃が可能。WPA3のSAEはその仕組みを変えて、オフライン攻撃を原理的に不可能にした。

02

WPA3の改善点と限界

PMF必須化と前方秘匿性を実現。ただしSAEでも能動的な繰り返し推測は可能（RFC 7664 §4）。OWEはAPの認証を提供しない（RFC 8110 §7）。

03

エンタープライズ移行

802.1X環境ではRADIUSサーバの変更は一般的に不要。Transition Modeで新旧デバイスを共存させながら段階的に移行できる。

この理解で次にできること

→

「なぜWPA3に移行すべきか」を認証方式の原理（SAEのオフライン攻撃耐性）から上長に説明できる

→

移行計画策定時に「デバイスのWPA3対応状況」と「Transition Modeの期間設定」を調査項目として設計できる

→

ゲストSSIDのOWE化について、「無線区間暗号化はできるがAP認証は提供しない」という効果と限界を踏まえて提案できる
