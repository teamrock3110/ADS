# Diagram Corp Eng

SaaS製品・Webサービス・ネットワーク技術の**仕組みを原理原則から**理解するための図解を生成する。
見る人は自分だけ。ゴールは「新規導入の提案・サービスの設定変更において、根拠を持って技術説明できる理解」。

**最重要の可視化原則**: 未知の技術を初めて理解するとき、全体のフローを先にASCIIアートで俯瞰する。
構造が頭に入ってから詳細に入る。設定値と実際の動作の対応関係を常に示す。

## コンテンツの方針

### 原理原則ファースト
- 表面的な使い方ではなく、**なぜそうなっているか**（Why）から説明する
- 「〇〇設定を変えると△△が変わる」の前に、「その設定が何を制御しているか」を先に説明する
- 省略して分かりにくくするより、丁寧に説明して腹落ちさせる

### 一次情報への徹底的なこだわり
- 説明の根拠となる公式ドキュメントURLを必ず示す
- 推測・憶測による説明は明示的に「※推測」とラベリングする
- 公式情報が見つからない部分は「公式情報なし」と明記する

### 関係性の可視化
- 単体の説明より、**コンポーネント間の関係・依存・フロー**を重視する
- 「Aが変わるとBとCに影響する」の全体像を先に見せる
- フェイルオーバー・バックアップ・冗長化のような「いざというとき何が起きるか」を図で表現する

## ワークフロー

### Phase 0: 公式ドキュメント調査（サブエージェント）

**exploreサブエージェント**を起動して、公式情報を徹底収集する。

```
Task({
  subagent_type: "explore",
  description: "公式ドキュメントの調査",
  prompt: `
    以下の技術・製品について、**公式情報のみ**を徹底調査してください：
    【対象】{トピック名}
    
    調査項目：
    1. 公式ドキュメントのURL（製品ベンダー・RFC・IETFなど）
    2. この技術が解決する問題（公式の説明）
    3. 主要コンポーネントとその役割（公式の定義）
    4. コンポーネント間の依存関係・通信フロー
    5. 設定値・パラメータとその意味（公式ドキュメントから）
    6. 障害・フェイルオーバー時の動作（公式ドキュメントから）
    7. よくある誤解・注意点（公式ドキュメントやFAQから）
    
    ※ 推測や二次情報（ブログ・Qiita等）は使わず、公式を優先してください。
    ※ 公式情報が見つからない項目は「公式情報なし」と明記してください。
    
    結果は項目ごとに「情報源URL」を添えて返してください。
  `
})
```

### Phase 1: 情報の整理と構造把握

1. URLが渡された場合は `WebFetch` で公式ページを取得
2. Phase 0のリサーチ結果を精査する
3. 以下の観点で情報を整理する：

| 観点 | 内容 |
|-----|------|
| 問題 | この技術が解決しようとしている問題は何か |
| 構成要素 | 主要なコンポーネントは何か |
| 関係性 | コンポーネント同士はどう繋がっているか |
| フロー | 通常時・異常時の動作フローはどうなっているか |
| 設定 | 動作を変える設定パラメータは何か |
| 根拠 | 各情報の公式ソースURL |

### Phase 2: 「まず理解する構造」を決める

技術の理解に最も重要な「骨格」を3つ選ぶ。

| 優先度 | 選定基準 |
|-------|--------|
| 高 | これを知らないと他の説明が理解できないもの |
| 中 | 理解の精度を上げる補足情報 |
| 低 | 応用・例外ケース |

### Phase 3: 図解の設計

**一枚絵サマリーは必ずASCIIアートで作る。** 全体像を文字で俯瞰できることが最優先。
その後、各セクションの詳細でHTML図（フロー図・比較図・シーケンス図）を使う。

| トピックの性質 | 一枚絵サマリー（ASCII） | 詳細セクション（HTML図） |
|-------------|----------------------|----------------------|
| 仕組み・動作フロー | 登場人物と矢印で流れを表現 | ステップフロー図 |
| コンポーネント間の関係 | ボックスと線で構成を表現 | 構成図（アーキテクチャ図） |
| 設定値と動作の関係 | 設定→判定→結果の流れ | 対応表 + フロー |
| 正常時 vs 障害時の比較 | 2つのフローを並列表示 | 左右対比図 |
| 認証・通信のシーケンス | 参加者と矢印でシーケンス | シーケンス図 |
| 階層・包含関係 | インデントで包含を表現 | 入れ子ブロック図 |

**詳細** → [references/ascii-art-guide.md](references/ascii-art-guide.md)
**HTML実装** → [references/html-structure.md](references/html-structure.md)

### Phase 4: HTML生成

以下の構造で生成する：

```
1. ヘッダー
   └─ タイトル + カテゴリバッジ + 一言の答え

2. 「ざっくり一言」【必須・30字以内】
   └─ 問いに対する本質的な答えを一文で先に示す
   └─ 例: 「WPA3は"パスワード盗み見攻撃"に強い認証方式に刷新した規格」
   └─ 読んだ瞬間に「何の話か」が分かる粒度にする

3. 「この図解でわかること」（冒頭サマリー）
   └─ 読み終わった後に何が分かるかを先に示す

4. 一枚絵サマリー【必須・ASCIIアート】
   └─ 全体像をASCIIアートで俯瞰する
   └─ 登場人物・コンポーネント・通信フローを文字で表現
   └─ 「これを見れば全体の構造が分かる」1枚

5. まず理解する3つの構造
   └─ 骨格となる知識を優先度順に

6. 各セクション詳細
   └─ 原理原則の説明
   └─ コンポーネント・フロー図（HTML）
   └─ 設定との関係（該当する場合）
   └─ 根拠バッジ（公式ドキュメントリンク）【各説明の直後にインラインで配置】

7. まとめ
   └─ 理解のポイントを3点で整理
   └─ 「この理解を根拠に〇〇できる」ゴール確認
```

**禁止事項**:
- 絵文字を使わない（`→` `✓` `↗` などテキスト記号はOK）
- React・shadcn/ui を使わない
- インタラクティブ要素を入れない（トグル・アニメーション等）
- `<style>` タグを追加しない（スタイリングはTailwindクラスで行う）
- `<script>` タグを追加しない（テンプレートに含まれるもの以外）
- 推測による説明を事実として書かない
- 一枚絵サマリーでASCIIアートを省略しない（HTML図だけで代替しない）
- 「ざっくり一言」を省略しない
- 末尾に「参照一次情報リスト」セクションを独立して置かない（インライン出典に一本化）
- 英語略語を補足なしで使わない（ただし閾値を守る → 下記参照）

**詳細** → [references/html-structure.md](references/html-structure.md)

### Phase 5: 批判的レビュー（サブエージェント）

**「自分が設定変更をする判断者」の視点**でレビューする。

```
Task({
  subagent_type: "generalPurpose",
  description: "図解の批判的レビュー",
  readonly: true,
  prompt: `
    あなたは「動画配信プラットフォーム会社のコーポレートエンジニア」です。
    社内システム・SaaSの導入設定・運用・業務効率化が主な業務で、
    IT全般の基本知識はあるが、専門的な技術は原理原則から理解しないと
    自分の言葉で説明できません。
    この図解の目的は「自分が設定変更をするとき、その根拠と設定後の状態を
    自分で確信し、上長に求められたら正しさを説明できること」です。
    
    以下のHTML図解を読んで、**忖度なく**批判してください：
    
    【HTMLファイルパス】{path}
    
    ## レビュー観点
    
    1. **原理原則の説明不足**: 「なぜそうなっているか」の説明が薄い箇所はないか
    2. **関係性の不明瞭さ**: コンポーネント間の依存・影響が分かりにくい箇所
    3. **一次情報の欠如**: 根拠URLなしで断言している説明はないか
    4. **設定変更のイメージ**: この図解を読んで「どの設定値を変えると何が変わるか」
       「変更後の状態はどうなるか」が具体的にイメージできるか。
       「なぜその値にするか」の根拠まで読み取れるか。
    5. **「自分の判断を上長に説明できるか」**: この図解を根拠に自分が設定変更した後、
       「なぜその設定にしたのか」「それで何が変わるのか」を上長に問われたとき、
       自信を持って答えられるか。「提案」ではなく「判断の事後説明」として耐えうるか。
    
    ## 出力形式
    
    ### 致命的な問題（これがあると判断の根拠にならない）
    - ...
    
    ### 改善すべき点（直せばもっと信頼できる図解になる）
    - ...
    
    ### 良かった点（このまま残すべき）
    - ...
    
    ### 総評
    5段階評価: ⭐⭐⭐☆☆（3/5）のように
    一言: 「この図解を読んだ後、〇〇の設定変更を自信を持って実行できる/できない、
         なぜなら△△。上長に設定根拠を問われたとき△△だから説明できる/できない。」
  `
})
```

### Phase 6: ブラッシュアップ

Phase 5のレビュー結果を踏まえてHTMLを修正する。

**優先度**:
1. 致命的な問題 → 必ず修正
2. 改善すべき点 → 可能な限り対応
3. 良かった点 → 維持

**よくある修正パターン**:
- 「なぜ」の説明を追加する
- 根拠バッジ（公式ドキュメントリンク）を追加する
- コンポーネント間の矢印・依存関係を明示する
- 「推測」ラベルを追加して公式情報と区別する

### Phase 7: デプロイ

```bash
# 1. output/ ディレクトリ確認・作成
mkdir -p output

# 2. HTMLファイルを output/{スラッグ}.html に保存済みであることを確認

# 3. surge.shにデプロイ
cd output && npx --yes surge {スラッグ}.html --domain diagram-{スラッグ}.surge.sh
```

スラッグはトピックに関連する短い英単語（例: `wifi-failover`, `saml-flow`, `okta-scim`）。

### Phase 8: 完了報告

```
完成・公開完了: 【図解タイトル】

【この図解でわかること】
（1〜2文で要約）

公開URL:
https://diagram-{スラッグ}.surge.sh

【根拠とした一次情報】
- （主要な公式ドキュメント名 + URL）

【この図解を根拠に提案できること】
- （3点）
```

## 英語略語の補足ルール

### 閾値: 「30歳のコーポレートエンジニアが調べないと分からない用語のみ」

**補足する（調べないと分からない）**:
- `EAPOL`（EAPをLAN上で運ぶフレーム形式）
- `GTK`（ブロードキャスト用グループ鍵）
- `ANonce` / `SNonce`（APまたはクライアントが生成した使い捨て乱数）
- `KRACK`（鍵の再インストール攻撃: Key Reinstallation Attack）
- `MIC`（改ざん検知コード: Message Integrity Code）
- `Dragonfly`（楕円曲線ベースのパスワード認証鍵交換プロトコル）
- `CNSA`（米国政府推奨の高強度暗号スイート）

**補足しない（体か本文中で説明する）**:
- `PSK`, `SAE`, `PMF`, `OWE` → 本文中で「〇〇（日本語の意味）」と最初の登場時に説明する
- `RADIUS`, `802.1X`, `EAP`, `MITM`, `AES` → コーポレートエンジニアが知っている前提の用語

### 補足フォーマット

**バッジ（`.abbr-tag`）は使わない。** インラインの括弧テキストで補足する。

```html
<!-- 良い例: 括弧テキストで軽く補足 -->
<strong>EAPOL（EAPをLAN上で運ぶフレーム形式）</strong>
<strong>GTK（ブロードキャスト用グループ鍵）</strong>
<strong>KRACK（鍵の再インストール攻撃）</strong>

<!-- 悪い例: バッジCSSは使わない -->
<strong>SAE<span class="abbr-tag">...</span></strong>  ← 禁止
```

本文中で説明済みの用語（PSK、SAEなど）は、2回目以降の登場で括弧補足も不要。

## 一次情報の引用ルール

### 基本原則: 「読者が疑う箇所にだけ URL を置く」

URL を全ての主張に貼るとノイズになり、重要な根拠が埋もれる。
**「この URL がなければ読者は判断できないか？」を自問してから置く。**

#### URL が必要な条件（どれか1つでも当てはまれば置く）
- 具体的な数値・仕様値を断言している
- 読者にとって意外な事実
- 設定変更の判断に直接影響する重要な制約

#### URL 不要な条件
- 製品カテゴリの一般説明（「Cursor は VS Code ベースのコードエディタ」）
- 業界で広く知られた事実
- 既に URL 付きで示した主張の補足・言い換え

**目安: 1つのカード・ステップにつき 0〜2 個程度。**

### 置き場所: 必ず主張の直後にインラインで

末尾に「参照した一次情報」セクションを独立して作ることを**禁止**する。
カード末尾にURLをまとめることも**禁止**（主張との1:1対応が取れなくなる）。
出典は対応する主張の**直後にインラインで**配置する。

```html
<!-- 良い例: 数値の断言の直後にインライン出典 -->
<p class="step-body">
  有料プランのコンテキストウィンドウは <strong>200K トークン</strong>。
  <a href="https://support.claude.com/.../8606394" target="_blank" class="source-link" style="font-size:0.75rem">
    ↗ Claude コンテキスト公式ヘルプ
  </a>
</p>

<!-- 悪い例: 全主張にバッジを貼る / 末尾リストにまとめる -->
<div class="section-card">
  <ul>...</ul>  ← 禁止
</div>
```

## 品質チェックリスト

### 必須

- [ ] 「ざっくり一言」が冒頭に30字以内で書かれている
- [ ] 一枚絵サマリーがASCIIアートで作られている
- [ ] 英語略語の補足は「30歳が調べないと分からない用語のみ・括弧テキスト」で記載
- [ ] 末尾に独立した「参照一次情報リスト」セクションがない（インライン出典に一本化）
- [ ] 「なぜそうなっているか」（原理原則）の説明がある
- [ ] コンポーネント間の関係・依存が図で示されている
- [ ] 根拠 URL は「数値の断言・意外な事実・判断に影響する制約」の箇所のみに絞っている（1カード 0〜2 個が目安）
- [ ] 一般的に知られている事実・製品説明に URL を貼っていない
- [ ] 推測による説明は「※推測」とラベリングされている
- [ ] 「まず理解する3つの構造」がある
- [ ] まとめに「この理解で〇〇できる」があること

### デザイン

- [ ] ADS配色のTailwindクラスを使用（html-structure.md参照）
- [ ] Lucide Iconsを適切に使用している（絵文字は禁止）
- [ ] スマホでも読みやすいレスポンシブ対応
- [ ] セクション・カードで視覚的にメリハリがついている
