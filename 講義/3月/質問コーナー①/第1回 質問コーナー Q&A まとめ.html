01

Cursor vs Claude Code の使い分け

どのような用途で使い分けていますか？

Cursor ── 7割
  • コンテンツを作りながらツールを磨く作業に最適
  • 仮想環境やSlack連携が便利
  • 初心者はまずCursorがおすすめ
プログラミング以外の用途にも強い
Claude Code ── 3割
  • アプリ開発が決まっていればこちら
  • プログラミングに最適化された設計
  • Web/スマホアプリを最初から作る時に使う
プログラムを作ることに特化
マコのひとこと

「どちらかが圧倒的に良いということはなくて一長一短あります。初心者のうちはCursorでいいんじゃないのかというのが私の理解です」

スキルの育て方・管理方法

スキルを育てるサイクル

1

2

マコのひとこと

「どんなツールを使おうともSSoTの原則を厳守するということは変わりません。いいツール・いいやり方は常に変わるけど、SSoTは守るべきということは変わりません」

実践ルール

最初は1事業部1リポジトリで始める。機密情報は別リポジトリに分離。大きくなりすぎたら適宜切り出す。全社共通情報はソース元フォルダから自動配信する。

現実:

プロサッカー選手に「絶対うまくなる1つのコツは？」と聞いたら「練習することだ」としか答えようがない。全12回のプログラムを通じて、いろんな切り口で伝え続けるのがこの講義の設計。

デザイン崩れ防護策

AIに微修正を頼むとスタイルが崩れる。どう防ぐ？

答え

Git + パッチワーク禁止

こまめにコミット。修正は全体を見て行う。

NG: パッチワーク修正
  • 「ここもっと右」「この色変えて」の繰り返し
  • AIが部分だけに集中し全体を見失う
  • 修正するたびに他の場所が壊れる
OK: プロの仕事
  • 「プロのUI/UXデザイナーとして仕事して」
  • 修正後のUIをアスキーアートで事前確認
  • こまめにGitでコミットして戻れるように
マコのひとこと

「一番良くないのはパッチワークを続けること。パッチワークを繰り返していくとAIも何に集中していいかわかんなくてどんどん性能が落ちていきます」

AIとセキュリティ

どこまで情報を入れていいのか？

情報の種類別 判断ガイド

一般的な業務情報・公開情報
基本的にAIに入力OK
顧客データ・営業資料
個人情報をサンプルデータに置換 / 社名を伏せて入力
個人情報・機密情報そのまま
名前・メール・住所・電話番号をそのまま入力しない
マコのひとこと

「危険か危険じゃないかという二元論ではなく、個別具体の情報でこれは良いのか悪いのかを判断していくのが大事。守るものが多ければより保守的に、少なければ車は危ないから乗らないという意思決定をする必要はない」

大事な前提

ほとんどのAIツールは学習に使うかどうかの設定ができる。また大手AI企業は個人が特定されない形に加工していると各社は言っている。ただし詳しくは必ず自分で確認すること。会社に所属している場合は会社のルールに従う。

ドリルの進め方

ドリル後半が難しくて立ち止まってしまう。進めて大丈夫？

不安: 全部理解しないと先に進めない

結論: 大丈夫です

ドリルは何度も薄く重ねる設計。1回ですべて理解できることはない。前に進み続けることで自然と復習もでき、知識が繋がっていく。

一番大事なこと

ドリルはあくまでドリル。実際に自分の作りたいものを作るという「自分で作ってみる」に勝る学びは存在しません。

マコのひとこと

「何度も薄く重ねていってまだわかっていないんだけどなって思っても、前に進み続けることで自然と復習もできて応用に進んでいけるように設計しています」
